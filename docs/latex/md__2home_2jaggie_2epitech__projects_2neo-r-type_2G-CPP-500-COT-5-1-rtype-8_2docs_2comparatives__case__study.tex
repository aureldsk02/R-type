Algorithms, Data Structures and Design Patterns}}\label{md__2home_2jaggie_2epitech__projects_2neo-r-type_2G-CPP-500-COT-5-1-rtype-8_2docs_2comparatives__case__study_autotoc_md93}

\DoxyHorRuler{0}
\hypertarget{md__2home_2jaggie_2epitech__projects_2neo-r-type_2G-CPP-500-COT-5-1-rtype-8_2docs_2comparatives__case__study_autotoc_md95}{}\doxysubsubsection{\texorpdfstring{Core Structures}{Core Structures}}\label{md__2home_2jaggie_2epitech__projects_2neo-r-type_2G-CPP-500-COT-5-1-rtype-8_2docs_2comparatives__case__study_autotoc_md95}

\begin{DoxyItemize}
\item \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}{\ttfamily std\+::vector\texorpdfstring{$<$}{<}std\+::optional\texorpdfstring{$<$}{<}T\texorpdfstring{$>$}{>}\texorpdfstring{$>$}{>}} for ECS components\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}

This structure allows efficient storage of components associated with entities. ~\newline
 Using {\ttfamily std\+::optional} provides direct index-\/based access while supporting empty slots (for deleted or missing entities) without requiring explicit identifiers. ~\newline
 This design reduces memory overhead, avoids redundant entity IDs, and keeps data contiguous in memory — significantly improving cache locality and sequential access performance.
\item \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}{\ttfamily std\+::unordered\+\_\+map} for component table storage\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}

Component tables are stored in {\ttfamily std\+::unordered\+\_\+map} containers to provide amortized constant-\/time ({\ttfamily O(1)}) access for insertion, lookup, and removal. ~\newline
 This is ideal for ECS systems where direct access patterns dominate, as it minimizes CPU cache misses. ~\newline
 An alternative would be {\ttfamily std\+::map} (a balanced binary tree), but it introduces logarithmic complexity ({\ttfamily O(log n)}), which is unnecessary in most real-\/time systems.
\end{DoxyItemize}

\DoxyHorRuler{0}
\hypertarget{md__2home_2jaggie_2epitech__projects_2neo-r-type_2G-CPP-500-COT-5-1-rtype-8_2docs_2comparatives__case__study_autotoc_md97}{}\doxysubsubsection{\texorpdfstring{Client-\/side Change Detection Algorithm}{Client-\/side Change Detection Algorithm}}\label{md__2home_2jaggie_2epitech__projects_2neo-r-type_2G-CPP-500-COT-5-1-rtype-8_2docs_2comparatives__case__study_autotoc_md97}
When a client receives a {\bfseries{snapshot}} from the server (a complete or partial state of all entities), it must determine\+:
\begin{DoxyItemize}
\item which entities are {\bfseries{new}} (present in the snapshot but not locally),
\item which entities are {\bfseries{deleted}} (missing from the snapshot but still locally stored),
\item and which entities must be {\bfseries{updated}} (present in both but with changed components).
\end{DoxyItemize}

To achieve this efficiently, a {\bfseries{hashmap-\/based diff detection algorithm}} is used\+:
\begin{DoxyItemize}
\item Each entity is indexed by its unique identifier ({\ttfamily entity\+\_\+id}).
\item The client maintains a local dictionary ({\ttfamily unordered\+\_\+map\texorpdfstring{$<$}{<}entity\+\_\+id, Entity\+Snapshot\texorpdfstring{$>$}{>}}).
\item When a new snapshot arrives\+:
\begin{DoxyItemize}
\item The snapshot is iterated to detect {\bfseries{new or modified}} entities by comparing component hashes against the local version.
\item The local state is then scanned to identify {\bfseries{removed}} entities missing from the new snapshot.
\end{DoxyItemize}
\end{DoxyItemize}

This approach yields an average complexity of {\bfseries{O(n)}} (linear with the number of exchanged entities), which performs efficiently under real-\/time update conditions, even with frequent incremental changes.

This approach provides several advantages\+:
\begin{DoxyItemize}
\item {\bfseries{Encapsulation of complexity}} — entity construction logic (and component dependencies) is hidden from the rest of the system. ~\newline

\item {\bfseries{Consistency}} — all entities of the same type are guaranteed to have the same component configuration. ~\newline

\item {\bfseries{Maintainability}} — new entity types can be added without modifying the ECS core. ~\newline

\item {\bfseries{Scalability}} — different factories can be defined for gameplay modules (e.\+g., player, enemies, projectiles, bosses).
\end{DoxyItemize}

An alternative approach could be to define entities through {\bfseries{configuration files}} or {\bfseries{prefabs}}, but using a factory allows better compile-\/time safety and avoids runtime parsing overhead during gameplay.
